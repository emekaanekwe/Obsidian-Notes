## Question 1: Motivation for ADTs

**(a)** Two reasons for using ADTs:

1. **Abstraction/Encapsulation** — ADTs separate the _what_ (the interface and operations) from the _how_ (the implementation), allowing programmers to use a data structure without needing to know its internal details. This reduces complexity.
2. **Reusability and interchangeability** — A well-defined ADT can be implemented in multiple ways (e.g., a Stack using an array or a linked list) without changing how it is used, making code more modular and maintainable.

**(b)** A **Dictionary** is ideal for a **student record system** — for example, mapping student IDs (keys) to their personal details and grades (values). Lookups by ID are direct and efficient, and there is a clear one-to-one key-to-value relationship, which is exactly what a Dictionary is designed for.

**(c)** A **Tree** is more appropriate than a Graph when representing a **file system directory structure**. Directories contain subdirectories and files in a strict parent-child hierarchy with no cycles and exactly one path between any two nodes. A graph's extra complexity (cycles, multiple paths) is unnecessary and would misrepresent the structure.

## Question 3: Graph Concepts

**(a)** A graph is **connected** if there exists a path between every pair of vertices — that is, no vertex is isolated or unreachable from any other vertex.

**(b)** A **directed acyclic graph (DAG)** is a directed graph that contains no cycles. All edges have a direction, and it is impossible to start at any node and follow edges back to the same node.

**(c)** A **cycle** is a path in a graph that begins and ends at the same vertex, passing through at least one other vertex, without repeating any edge. In other words, you can "loop back" to where you started by following edges.

**(d)** A **subgraph** is a graph formed from a subset of the vertices and edges of a larger graph, where every edge in the subgraph connects vertices that are also in the subgraph.

## Question 9: Robot Vacuum

**(a)** Formal FSM notation:

```
States:      Q = {Idle, Cleaning, Docked}
Alphabet:    Σ = {start, finished, battery_low, recharged}
Start state: Idle
Transitions: δ:
  δ(Idle,     start)       = Cleaning
  δ(Cleaning, finished)    = Idle
  δ(Cleaning, battery_low) = Docked
  δ(Docked,   recharged)   = Idle
```

**(b)** Final state: **Idle** (Idle →[start]→ Cleaning →[battery low]→ Docked →[recharged]→ Idle)

**(c)** An FSM makes all possible states and transitions **explicit and structured**. It is easier to verify correctness, extend with new states, and avoid logic errors compared to deeply nested if/else conditionals that become hard to read and maintain.

**(d)** An appropriate combination of two ADTs: **Graph + Priority Queue**.

**(e)** Explanation:

The **Graph** (specifically a state graph / FSM) models the vacuum's _operational modes_ — the rooms and their connections represent the physical space, and nodes/states capture whether the vacuum is Idle, Cleaning, or Docked in a given zone. The graph tracks which rooms have been cleaned and which are reachable, and the motion sensor constraint (only active when nobody is home) can be encoded as a condition on edge traversal.

The **Priority Queue** manages the _task scheduling_. Each cleaning task (vacuum main room, edge cleaning, spot cleaning, etc.) is enqueued with a priority value — for instance, the main room may have higher priority than spot cleaning. The vacuum always dequeues and executes the highest-priority pending task next, ensuring important tasks are not delayed by lower-priority ones.

These two ADTs **interrelate** because the Graph determines _which tasks are accessible_ (e.g., you can't clean the bedroom if the path to it is blocked or the vacuum is docked), while the Priority Queue determines _which accessible task to do next_. The Graph feeds the Priority Queue by identifying available tasks from the current state, and the Priority Queue feeds back into the Graph by directing which node (room/task) to transition to. Neither ADT alone is sufficient: the Graph lacks task prioritisation, and the Priority Queue lacks spatial/state awareness.
