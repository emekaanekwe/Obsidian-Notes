# FIT5120 — Week 1 Prep Templates & Step-by-Step Guide

> [!info] How to use this file
> This is one master file. Once your semester starts, split each `##` section below into its own Obsidian note (right-click heading → "Extract to new note") so each template lives independently and can be linked (`[[Problem Statement]]`, `[[Empathy Map]]`, etc.) from your weekly project notes. Do the drills below **in this order** — each one builds on the last.

---

## Step-by-step order of operations (do this over ~5 sessions)

- [ ] **Session 1** — Fill in the Problem Statement template using a dummy topic
- [ ] **Session 2** — Build one full Empathy Map + Persona from that dummy topic
- [ ] **Session 3** — Write 5 User Stories with Gherkin acceptance criteria for that persona
- [ ] **Session 4** — Practice one Reflective Diary entry using a real past experience
- [ ] **Session 5** — Set up your Meeting Minutes / Project Governance log structure (empty, ready to use week 1)

Pick **one dummy topic** and carry it through Sessions 1–3 so the outputs connect (this mirrors exactly how the real project will flow: problem → empathy → persona → user stories). Suggested dummy topic if you want one: *"international students at Monash struggling to find affordable, healthy food near campus."* Swap for anything you like — the practice matters more than the topic.

## 1. Problem Statement Template

> [!note] What this is for
> This is graded material — Assignment 1A (Proposal to Industry Mentor) requires a 4–5 sentence, evidence-backed problem statement. Practicing the format now means week 4 is just refinement, not a cold start.

### Instructions
1. State the problem in **one plain sentence** — no jargon, no solution baked in.
2. Say **who** is affected and **how badly** (scale, frequency, severity).
3. Cite **one piece of evidence** — a stat, a news article, or a credible report. Not an opinion.
4. Explain **why it matters now** — urgency, trend, or consequence of inaction.
5. End with **what "solved" would look like**, in outcome terms — not feature terms.

### Template

```markdown
## Problem Statement — [Project/Topic Name]

**Who is affected:** 
**What is the problem (plain language):** 

**Evidence it's real:**
- Source: 
- Key stat/quote (your own words, not copied text): 
- Link: 

**Why now / why it matters:**


**What "solved" looks like (outcome, not feature):**


**Draft 4–5 sentence version:**
> [Write the flowing paragraph version here — this is what you'd say in a pitch.]
```

### Self-check before moving on
- [ ] Could a non-technical person understand this in 10 seconds?
- [ ] Is there a citation, not just an assertion?
- [ ] Have I avoided naming a solution/technology in the problem statement itself?

## 2. Empathy Map Template

> [!note] What this is for
> Empathy maps force you to separate observable behaviour from assumption. You'll build one per key user type before writing personas.

### Instructions
1. Pick **one specific user**, not "users" in general (e.g. "a first-year international student living off-campus," not "students").
2. Fill each quadrant using **things they'd actually say or do** — avoid guessing feelings without a behavioural anchor.
3. Do the "Pains" and "Gains" sections **last** — they should emerge from the other four quadrants, not be invented first.

### Template

```markdown
## Empathy Map — [User Type]

**SAYS** (direct quotes or paraphrased things this person would say)
- 
- 

**THINKS** (what's on their mind — may differ from what they say)
- 
- 

**DOES** (observable actions/behaviours)
- 
- 

**FEELS** (emotional state — inferred from says/thinks/does, not invented)
- 
- 

**PAINS** (frustrations, obstacles, risks they face)
- 
- 

**GAINS** (what would make their life better — wants, needs, measures of success)
- 
- 
```

## 3. Persona Template

> [!note] What this is for
> Turns the empathy map into a reference character your team can design against. Keep it one page — personas that sprawl stop being useful.

### Instructions
1. Give them a name and a photo placeholder (a stock image is fine) — this makes the persona easier for the team to reference in discussion.
2. Base every field on the empathy map — don't add new invented traits.
3. Write **one goal** and **one frustration** that directly connect to your problem statement.

### Template

```markdown
## Persona — [Name]

**Photo:** [placeholder]
**Age / Role / Context:** 
**One-line summary:** 

**Goals:**
- 

**Frustrations:**
- 

**A day in their life (2-3 sentences):**


**Quote that captures them:**
> "..."

**Relevant tech/behaviour habits:**
- 
```

## 4. User Story + Gherkin Acceptance Criteria Template

> [!note] What this is for
> This is the format your team will use to define what gets built, sprint by sprint. Acceptance criteria are what QA/testing checks against — vague ones cause rework later.

### Instructions
1. Write the story in the standard three-part form.
2. Keep it to **one user action**, not a bundle of features.
3. Write acceptance criteria in Given/When/Then — at least 2 per story (a "happy path" and one edge case).

### Template

```markdown
## User Story — [ID: US-01]

**As a** [type of user],
**I want** [goal/action],
**so that** [reason/benefit].

### Acceptance Criteria (Gherkin format)

**Scenario 1: [happy path]**
- Given [initial context]
- When [action taken]
- Then [expected outcome]

**Scenario 2: [edge case]**
- Given [initial context]
- When [action taken]
- Then [expected outcome]

**Priority:** [Must / Should / Could — MoSCoW]
**Estimate:** [story points, fill in once team calibrates scale]
```

### Practice drill
Write 5 of these for your dummy persona before moving on. Vary priority — not everything is "Must."

## 5. Reflective Diary Template

> [!note] What this is for
> Individually assessed, submitted regularly across the semester (~10% + contributes to Learning Journey). The structure is fixed: Description → Evaluation → Analysis → Action Plan. Practicing this now means you're not inventing the format under deadline pressure in week 1.

### Instructions
1. Pick a **specific incident**, not a general summary of the week.
2. Be honest about the "felt at the time" section — vague or sanitised entries read as inauthentic and score worse.
3. The Action Plan must be **concrete and forward-looking** — "I'll communicate better" is too vague; "next time I'll flag disagreement in the moment rather than after the meeting" is usable.

### Template

```markdown
## Reflective Diary — [Date / Week #]

**Event/Incident (what happened, briefly, including your own and others' actions):**


**How I felt at the time:**


**Evaluation — what was most satisfying and/or troubling about it:**


**Analysis — why do you think it happened? What else could you have done?**


**Action Plan — what will you do differently next time? What are the implications going forward?**

```

### Practice drill
Write one entry now using a **real past incident** — e.g. your FIT5215 grading dispute, or a past group project moment. This gets the format into muscle memory before it's graded material.

## 6. Meeting Minutes / Project Governance Log Template

> [!note] What this is for
> Reviewed every class from week 1. Teams that start this on day one look organised immediately; teams that build it reactively in week 4 are visibly scrambling. Set the empty structure up now.

### Instructions
1. One note per meeting, named consistently: `YYYY-MM-DD Meeting Minutes`.
2. Keep action items in a **separate running log** (template below) so they're trackable across weeks, not buried in individual meeting notes.
3. Assign an owner and due date to every action item — no owner means it won't get done.
